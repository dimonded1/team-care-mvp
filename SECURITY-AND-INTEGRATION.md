# Безопасность и подготовка Bitrix

Дата проверки: 23 июля 2026 года.

## Исправлено

- Docker-образы Node.js и non-root nginx закреплены по immutable SHA-256 digest, чтобы сборка не подхватывала незаметно изменившийся образ по плавающему тегу.
- Runtime по-прежнему работает без root-прав, с read-only filesystem, healthcheck и защитными HTTP-заголовками из существующей nginx-конфигурации.
- Исправлено восстановление результата мэтчинга: сохранённый подопечный больше не получает проценты и объяснения, рассчитанные для другого животного.
- Bitrix-адаптер вынесен в отдельный server-only каталог и выключен по умолчанию.
- Включённый адаптер отклоняет HTTP webhook, незаполненные ID и отсутствующий секрет.
- Секрет интеграции не входит в публичный объект конфигурации, не отправляется в Bitrix и сравнивается через timing-safe проверку.
- Создание заявки на опеку требует явного согласия.

## Bitrix: что уже готово

- Один блок переменных: `.env.bitrix.example`.
- Маппинг заявки опекуна в CRM lead.
- Маппинг события заботы в элемент smart process.
- Отдельная авторизация будущего серверного endpoint.
- Тесты выключенного режима, валидации, маппинга и отсутствия утечки секрета.

## Что нужно получить перед включением

- HTTPS webhook URL Bitrix24.
- ID формы/сущности и всех пользовательских полей.
- Отдельный случайный server-only секрет для входящих запросов приложения.
- Решение о сервере, который будет принимать запрос от браузера: GitHub Pages не может безопасно хранить эти значения.

До этого момента `BITRIX_ENABLED=false` должен оставаться неизменным. Клиентский код не импортирует каталог `integrations/bitrix`.

## Намеренно не изменено

- `.github/workflows/**` и любая конфигурация GitHub Actions/Pages — по условию задачи.
- Реальная отправка пользовательских данных — выключена до получения конфигурации, серверного окружения и финального состава согласий.

## Оставшиеся риски перед production

- Онлайн-проверка npm advisory не завершилась: среда выполнения не получила доступ к `registry.npmjs.org`. Локальный граф зависимостей целостен, тесты и production-сборка проходят, но перед релизом нужно повторить `npm audit --omit=dev` в среде с доступом к реестру.
- Нужен серверный endpoint с rate limiting, ограничением размера тела, схемой входных данных, журналированием без персональных данных и политикой удаления.
- Webhook Bitrix необходимо хранить только в секрет-хранилище сервера и регулярно ротировать.
- После появления реальной схемы Bitrix нужно повторить threat model и интеграционные тесты на тестовом портале.
